# Asset Gate

This is a quick scope-decision step the skill runs automatically. The goal is NOT to block production — it is to pick the right delivery mode so the result is both **complete** and **honest**.

Default bias: **generate aggressively**. Use placeholders only for things that genuinely cannot be honestly generated (real shipped product screens, real brand logos, real user quotes, real metrics). Never deliver an empty deck full of placeholder boxes.

The gate has three outcomes: **FULL_CASE_STUDY / VISUAL_PROPOSAL / CRITIQUE**. Pick one in one line and move on.

## Step 1 — Quick Asset Scan (do not interrogate the user)

Scan the workspace and the user's message. Mark each as `present` / `missing`. Do not ask the user to fill out a form — infer what you can.

```text
Real product UI screenshots: present | missing
Brand logo / brand color: present | missing
Project facts (name, role, scope, year, platform): present | missing | partial
User research / metrics / quotes: present | missing
```

If the project involves a real public brand or product, also run the verification flow in `asset-and-fact-protocol.md` once — do not block on it.

## Step 2 — Pick The Mode

| Condition | Mode | What the skill produces |
|---|---|---|
| Real screenshots present | **FULL_CASE_STUDY** | Full deck. Real screenshots in real frames. Generated images for covers, section covers, 3D props, scenes. |
| No real screenshots, project concept is clear | **VISUAL_PROPOSAL** | Full visually-complete deck. Generate cover, section covers, 3D props, AND concept UI screens via the image model. Label the deck as "设计提案 / Visual Direction Proposal" so reviewers know it is concept work. |
| User wants only feedback on existing work | **CRITIQUE** | No image generation. Page-by-page review. |

Notes:

- VISUAL_PROPOSAL is the default when the user gives a project idea but no screenshots. It is honest, complete, and image-rich. It is NOT a placeholder deck.
- A deck full of `[Real screenshot pending]` boxes is not an acceptable output. If you are about to ship that, switch to VISUAL_PROPOSAL and generate the screens as concept visuals instead.

## Step 3 — State The Mode In One Line

Before visual work, write one line so the reader knows the framing:

```text
Mode: VISUAL_PROPOSAL — no real product screenshots provided, deck will be delivered as a complete visual direction proposal with image-model-generated concept screens.
```

That is enough. Move on to step 4 of the Auto-Run sequence.

## Honesty Rules (non-negotiable, even in VISUAL_PROPOSAL)

- NEVER fabricate specific numbers ("DAU 12 万", "转化率提升 28%"). If the user did not provide them, omit the metric block or label it `[指标待填]`.
- NEVER invent user quotes, persona names, or research findings as if they were real.
- NEVER claim a real existing company / brand / app as the project subject without user confirmation.
- Concept UI screens generated by the image model are fine — they are clearly part of a design proposal, not evidence of a shipped product.
- Real brand logos that don't exist yet: use a wordmark in the deck's display face as a placeholder, not a fake logo.

## Page-Level Concept Badge (VISUAL_PROPOSAL mandatory)

In VISUAL_PROPOSAL mode, EVERY page that contains a non-real UI surface (image-model concept screen, HTML-rendered dashboard mockup, poster series, designed component board) MUST carry a small badge in the top-right or bottom-right corner of the page:

```text
[概念 / Concept]   ← preferred, bilingual, 11-13px, in the deck's accent color on a 1px outlined chip
```

Cover, contents, section covers, thanks pages do NOT need the badge. Real screenshots (in FULL_CASE_STUDY) do NOT need the badge.

The badge is the smallest and most important honesty mark in the deck. Without it, a reviewer flipping through the middle of the deck cannot tell concept work from shipped product. With it, the deck can be visually complete AND fully honest.

A reusable HTML+CSS snippet is provided at `vendor/design-assets/assets/concept_badge.html` — copy the markup and the required CSS block into every concept page rather than re-authoring it. Variant labels (`[ 概念数据 / CONCEPT DATA ]`, `[ 概念研究 / CONCEPT RESEARCH ]`, `[ 概念 · 设计提案 / DESIGN PROPOSAL ]`) are listed in that file for context-specific use.

## Placeholder Format (use sparingly)

When something genuinely must be a placeholder:

- Metric: `[指标待填]` instead of a fabricated number.
- Real user quote: omit, or label `[用户访谈待补]`.
- Real shipped product screenshot (when the project IS a real shipped product but the user didn't share screens): a single dark/paper panel with `[真实产品截图待补充]`.

Do NOT use placeholders for: covers, section covers, 3D props, atmospheric backgrounds, concept UI screens in a VISUAL_PROPOSAL deck. Generate those.
