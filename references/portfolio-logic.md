# Portfolio Logic Rules

## Core Principle

A strong UI/UX portfolio deck is a proof system. It persuades through a sequence of visible decisions, not through decorative polish alone.

Each project should make four claims:

- I understood the product context.
- I found the real design problem.
- I explored and chose a solution for a reason.
- I can ship polished screens and explain the result.

## Case Study Spine

Use this order unless the user's material strongly suggests another:

```text
1. Context
2. Goal
3. Problem
4. Research or benchmark
5. Insight
6. Design strategy
7. Structure or flow
8. Key iterations
9. Final interface
10. Interaction/details
11. Outcome
12. Reflection
```

## What To Show On Each Stage

Context:
State product type, platform, user group, business scenario, role, timeline, and responsibility. Keep it concrete.

Goal:
Use one clear objective. Example: improve homepage task entry efficiency, reduce confusing banner occupancy, improve dashboard decision value, or make first-use onboarding more emotionally aligned.

Problem:
Avoid vague claims like "experience is poor". Convert them into observable issues: redundant entry points, unclear hierarchy, long path, weak feedback, hidden core data, mismatched visual tone, low trust, or cognitive overload.

Research or benchmark:
Show method and finding, not just logos. For competitive analysis, capture 3-5 patterns and translate them into opportunities.

Insight:
Condense analysis into a design lever. Example: "Managers need trend clues before detailed numbers" or "First-time users need low-pressure setup, not a form dump."

Design strategy:
State 2-4 principles. Use verbs: reduce, surface, group, guide, emphasize, reassure, compare, automate.

Structure or flow:
Show IA, task flow, screen relationship, or data layer. This is where the work becomes interaction design rather than visual arrangement.

Key iterations:
Use before/after or option A/B. Every comparison needs a reason for the chosen option.

Final interface:
Show the complete page or flow first, then zoom into details. Let the viewer see both the product and the craft.

Interaction/details:
Show states, components, edge cases, empty states, chart rules, color semantics, or animation logic.

Outcome:
Use real metrics when available. If no metrics exist, use validation notes, design review result, handoff quality, stakeholder feedback, or next-step reflection.

## Logic By Project Type

Mobile app redesign:
Emphasize task path, first screen hierarchy, tab/navigation logic, onboarding, key components, and before/after flows.

Ecommerce or marketplace app:
Emphasize business model, merchant/user roles, conversion path, product-detail clarity, cart/order flow, payment confidence, promotion modules, and operational efficiency.

Data dashboard or large-screen visualization:
Emphasize information priority, data model, chart selection, map/layer logic, alert levels, operational scenarios, and real-time decision value.

Enterprise/admin product:
Emphasize role permissions, workflow density, status management, table/filter design, exception handling, and efficiency.

SaaS official website:
Emphasize value proposition, product trust, solution structure, conversion CTA, information hierarchy, brand credibility, and how the site explains the product.

Component/icon library:
Emphasize consistency, reuse, naming, state coverage, extensibility, developer handoff, and how the library reduces future design/development cost.

Operations visual design:
Emphasize campaign goal, audience, channel size, visual hook, promotion hierarchy, brand continuity, and conversion or engagement target.

Brand/visual identity appendix:
Emphasize system consistency: logo, typography, color, applications, spatial mockups, and how it supports product perception.

Interview case study:
Emphasize clarity over spectacle. Show role, constraints, process, tradeoffs, and what you would improve next.

## Anti-Patterns

- Do not show only final mockups without problem and reasoning.
- Do not use long paragraphs that read like a report.
- Do not use fake user quotes or fake metrics unless clearly labeled as illustrative placeholders.
- Do not let visual system pages appear before the viewer understands the product problem.
- Do not make every page the same layout rhythm.
- Do not turn competitor analysis into logo wallpaper.
- Do not bury the designer's specific role.

## Page-Level Writing Rules

- Use one headline claim per page.
- Use bilingual titles only when they add portfolio polish; keep English short and atmospheric.
- Prefer noun phrases for section labels and action phrases for findings.
- Keep body copy brief: 2-5 bullets or 40-120 Chinese characters per block.
- Convert abstract qualities into design moves. "Premium" becomes lighting, depth, contrast, spacing, type hierarchy, and restrained accent color.
