# Image-First Deck Workflow

This is the **primary delivery path** of this skill. The user gets an HTML deck where each page is a full-bleed image-model-generated visual (gpt-image-2 / Nano Banana Pro / Gemini 3 Pro Image / Imagen 4), with editable text and clickable navigation layered on top via HTML.

Use this workflow whenever the deliverable is a UI/UX portfolio deck. Do not fall back to "agent writes the whole UI in CSS" unless the user explicitly asks for that.

## Division Of Labour For In-Frame UI Screens

Verified 2026-05: `gpt-image-2` renders Chinese / CJK UI labels cleanly and on-brief. Treat the image model as the default for in-frame concept UI screens regardless of label language.

| In-frame UI label language | How to produce | Where defined |
|---|---|---|
| Chinese / English / any language at typical UI sizes | Generate inside the page background image via the image model (default) | This file, `in_image_ui_style` |
| Dense numeric dashboards (any language) | Hand-build in HTML/SVG | `references/dashboard-rendering.md` |
| Fallback when only a CJK-garbling model is available | Hand-build in HTML/CSS inside the device frame ON TOP of the image background | `references/html-ui-screen-lane.md` (deprecated, fallback only) |

The image-first workflow is the spine. Hand the image model the full page including the in-frame UI; only peel pages out to other lanes when the image model genuinely cannot render them credibly (dense data dashboards) or when the available image model is verified to garble CJK.

## Why This Workflow Exists

Two things have to be true at once for a portfolio deck:

1. **Visual quality must be image-model-grade.** Agents writing CSS portfolios produce AI slop. gpt-image-2 / Nano Banana Pro / Gemini 3 Pro Image produce magazine-grade visuals.
2. **Text and navigation must be editable.** A deck is not a poster. Titles, project names, contact info, and links must be changeable without re-generating images.

The image-first deck achieves both: the image model owns the visual surface; HTML owns the text and the navigation.

## Architecture

Each page = three layers:

```text
+------------------------------------------------------+
| Layer 1 — Background image (full-bleed, generated)   |
|   - 16:9 at 2048x1152 from gpt-image-2               |
|   - Includes scenery, 3D props, device frames,       |
|     AND in-frame concept UI screens                  |
|   - Has explicit reserved areas for text and links   |
|   - Contains NO editable text inside the image       |
+------------------------------------------------------+
| Layer 2 — HTML text + link layer (absolute-positioned)|
|   - Chinese title in real font (思源黑体 Heavy etc.) |
|   - Project name, year, role, contact                |
|   - <a> tags for clickable links                     |
|   - Positioned with % coordinates over Layer 1 slots |
+------------------------------------------------------+
| Layer 3 — deck_stage.js navigation                   |
|   - Page turning, keyboard nav, hash routing         |
|   - Letterbox + auto-scale to viewport               |
|   - Speaker notes, print-to-PDF                      |
+------------------------------------------------------+
```

**Concept UI screens go inside Layer 1.** The user has confirmed: phone/laptop screens are generated by gpt-image-2 as part of the page image, not pasted in later as separate screenshots.

## Hard Rules

1. Every deck page MUST be a single full-bleed background image generated by a real image model. No CSS-painted backgrounds for the main visual.
2. The background image MUST include explicit reserved areas (negative space) for text and links. Specify the reserved areas in pixel coordinates inside the prompt.
3. The image MUST contain NO editable Chinese title text rendered by the model. All title text lives in HTML over the image. (Exception: a single short English label baked into the image is OK if intentional.)
4. Concept UI screens INSIDE phone/laptop frames ARE generated by the image model (per user decision). They are concept work, labelled at the deck level as "设计提案 / Visual Direction Proposal" in VISUAL_PROPOSAL mode.
5. Every page prompt MUST start with the project's PADB (per `visual-consistency.md`) and pass the previously-generated cover as a reference image to maintain consistency.
6. The HTML uses huashu's `deck_stage.js` shell (per `huashu-vendor-usage.md`) for navigation. Do not reinvent the page-turning engine.
7. Links must be real `<a href>` elements, not painted in the image. Same for buttons that should look clickable.

## Workflow Steps

### Step 1 — Confirm Mode And Style

Run the Auto-Run sequence in SKILL.md. Land on:
- Mode: `FULL_CASE_STUDY` or `VISUAL_PROPOSAL` (default).
- Style direction: chosen via guided-style-selection.

### Step 2 — Write The PADB

Per `visual-consistency.md`. This is the visual contract for every page in this project.

The PADB for image-first deck must additionally specify:

```text
in_image_ui_style:      The visual language of concept UI screens rendered inside device frames.
                        e.g. "Clean dashboard with 8px grid, sans display, single accent color from PADB,
                              real-looking data viz (line/bar/area), no fake garbled text — use simple
                              lorem-style English labels like 'Revenue', 'Active Users' that are short and plausible."
device_lock:            iPhone 15 Pro (titanium black) for all phone shots; MacBook Pro 16 (matte aluminum) for all laptop shots.
in_image_text_policy:   No Chinese text inside images. English UI labels OK only if short, plausible, and from a fixed vocabulary.
text_safe_zones:        Standard reserved areas per page role (see Layer Map below).
```

### Step 3 — Define The Layer Map Per Page Role

Each page role has a standard reserved-area template. Use these as the default and customize per page.

| Page role | Reserved text/link zones (within 2048x1152) |
|---|---|
| Cover | Top-left 28% (title block: x=80–650, y=100–520). Bottom-left 20% (designer name + year + contact: x=80–650, y=900–1080). |
| Section cover | Center-left 35% (section number + Chinese title: x=80–800, y=380–720). |
| Project background | Left 38% (Chinese title + 2-3 paragraph body: x=80–820, y=120–950). Right 60% is hero image. |
| Problem diagnosis | Left 40% (numbered analysis: x=80–880, y=120–1000). Right 55% is annotated visual. |
| Core UI / concept screen | Bottom 18% (caption strip: x=80–1968, y=900–1080). The screen itself is in-image. |
| Component library | Right 35% (rationale + metrics: x=1280–1968, y=120–1000). Left is component board image. |
| Outcome | Center 60% (large title + 3 metric cards: x=400–1648, y=200–900). Metric numbers can be HTML text on top of generated card backgrounds. |
| Thanks / Contact | Center 50% (大字感谢 + contact link: x=512–1536, y=300–800). |

These coordinates go into BOTH the image prompt (as "reserved empty area") AND the HTML CSS (as absolute-positioned text layer location).

### Step 4 — Generate Cover First

Generate the cover (4 candidates) per the PADB + cover layer map. Pick one. This is now the project's visual reference image.

### Step 5 — Generate Two Showcase Pages

Generate 1 section cover and 1 core-UI page. Both prompts:
- Begin with the full PADB.
- Pass the chosen cover as reference image.
- Specify the reserved text zones from the Layer Map.
- For core-UI pages: explicitly describe the in-frame concept screen (what kind of dashboard / app, what layout, what plausible English labels).

Run the 5-axis consistency QA per `visual-consistency.md`. Show the user the 3 images side by side. Wait for confirmation.

### Step 6 — Build The HTML Skeleton

Use `templates/deck-skeleton.html` (created alongside this file). Each page is a `<section>` containing:

```html
<section data-page-role="cover">
  <img class="bg" src="generated-images/01-cover.png" alt="">
  <div class="text-layer text-layer--cover">
    <h1>作品集标题</h1>
    <p class="meta">2026 · 设计师姓名 · UI/UX</p>
    <a href="mailto:..." class="link">contact@example.com</a>
  </div>
</section>
```

The `<deck-stage>` web component from `vendor/huashu-design/assets/deck_stage.js` wraps all sections and handles navigation.

### Step 7 — Batch Generate Remaining Pages

Once showcase is confirmed:

1. Loop through the remaining page roles in the storyboard.
2. For each page: prompt = PADB + Layer Map for that page role + cover as reference image + page-specific subject/scene description.
3. Save outputs to `generated-images/` with predictable names (`02-section-cover-A.png`, `05-project-1-core-ui.png`, etc.).
4. Add a matching `<section>` to the HTML for each new image.

### Step 8 — Apply Text Layer

For each page, write the text layer with absolute positioning that matches the reserved zones. Use real Chinese display fonts. Wrap any link as `<a>`.

### Step 9 — Final 5-Axis QA + Open In Browser

Open the HTML deck in a browser. Run the 5-axis consistency QA across all background images one more time (open `generated-images/` folder). Spot any drift, regenerate offending pages with stronger PADB/reference-image alignment.

## Image Prompt Recipe (Image-First Specific)

Use this on top of the PADB:

```text
[PADB block]

--- PAGE-SPECIFIC INSTRUCTIONS (image-first deck) ---
page_role:              cover | section_cover | project_background | problem | core_ui | component | outcome | thanks
deck_position:          page X of Y
canvas:                 16:9 at 2048x1152
reserved_text_zones:    [paste exact zones from the Layer Map for this role]
                        These zones MUST contain only soft background gradient or low-contrast texture —
                        NO subjects, NO props, NO hard edges that would clash with overlaid text.
in_image_subject:       [describe what fills the rest of the frame]
in_image_ui:            [if a phone/laptop frame is in the shot, describe the in-frame concept UI:
                         layout, density, plausible English labels, chart types, accent color from PADB.
                         Forbid Chinese text, fake garbled text, random brand logos.]
device_frame:           inherit from PADB.device_lock
links_planned:          [list of clickable hot zones that will be overlaid as <a> tags in HTML —
                         reserve clean negative space for them too]

negative:               [PADB.forbidden_colors] [PADB.forbidden_props]
                        no Chinese text in image
                        no garbled text
                        no fake brand logos
                        no watermark
                        no AI sparkles
                        the reserved text zones must remain visually clean
```

## Failure Modes To Watch For

- **Image has Chinese text baked in** → the model misspelled it, or the text is in the wrong font. Regenerate with stronger negative constraint.
- **Reserved text zone got filled with props** → the model ignored the safe zone. Regenerate with the zone described in pixel coordinates AND as "must remain a soft, low-contrast area".
- **Concept UI inside the phone looks fake/AI-slop** → tighten the in_image_ui spec: real chart types (line/bar/area), plausible short English labels, real grid alignment, single accent color. Pass a strong reference UI image if available.
- **Page 1 and page 5 don't feel like the same deck** → PADB drift. Run 5-axis QA, regenerate the offender, never the whole set.
- **HTML text and image text both appear** → kill the text inside the image. The image is a stage, not a poster.

## Variant — Image-As-Page (OCR Overlay)

Verified 2026-05. For the highest-craft single pages (covers, case-study heroes, anything where typography composition IS the design) the layered approach above sacrifices `gpt-image-2`'s strongest capability: it will absolutely render Chinese typography, gradient titles, KPI tiles, and integrated 3D mockups in one pass at print quality. Splitting bg + overlay forces gpt-image-2 to leave reserved holes and HTML to fake the typography on top — losing the editorial feel.

In this variant, **the image IS the entire page including text**, and interactivity comes from invisible `<a>` hotspots positioned by OCR.

### Architecture

```text
+------------------------------------------------------+
| Layer 1 — Single full-page image (text + UI baked in)|
|   - 1792x1024 (16:9) from gpt-image-2                |
|   - Title, body, KPI numbers, contact, mockups all   |
|     rendered by the model in one composition         |
+------------------------------------------------------+
| Layer 2 — Transparent <a> hotspots                   |
|   - PaddleOCR extracts pixel-precise bbox per text   |
|   - HTML overlays a transparent <a> on each big      |
|     horizontal title; hover shows highlight+tooltip  |
|   - Filter: height >= 60px AND width/height >= 1.1   |
|     (skip small/diagonal/decorative text)            |
+------------------------------------------------------+
```

### Toolchain

```text
prompts/<page>.txt
   │
   ▼  scripts/image_client.py generate --model gpt-image-2 --size 1792x1024
generated-images/demo/<page>.png        (full page, text baked in)
   │
   ▼  scripts/ocr_hotspots_paddle.py --lang ch
generated-images/demo/<page>.hotspots.json   (PaddleOCR pixel bboxes)
   │
   ▼  scripts/build_interactive_page.py
generated-images/demo/<page>.html       (self-contained, double-click to open)
```

Optional verification: `scripts/overlay_hotspots.py` renders red bbox boxes over the source image to eyeball OCR alignment.

### When To Use This Variant Instead Of The Layered Workflow

- Cover pages where the title is the design (gradient typography, mixed CN/EN, decorative outline marks).
- Case-study hero pages with integrated mockups + KPI tiles + role chips composed as one editorial spread.
- Single "key visual" pages that need to look like a magazine layout, not a presentation slide.

Stick with the layered workflow (Steps 1–9 above) for: deck navigation, content-heavy text pages, dashboards, anything that requires copy-paste-able body text or rapid text edits.

### Hard Rules For This Variant

1. Image generated by `gpt-image-2` (default; accepts 2048x1152, ~1.2MB, ~80s render) via `/images/generations` on the pockgo relay. Opt up to `gpt-image-2-2k` only when the relay is healthy — the `-2k` variant has been intermittently overloaded with HTTP 524 in 2026-05 (verified with servio/case01 — 5 retries all 524 on `-2k`, first try success on plain `gpt-image-2`).
   - DO NOT use `gpt-image-2-4k` on this relay: origin > 120s, Cloudflare 524.
   - DO NOT use `nano-banana-pro` / `gemini-3-pro-image-preview` / `flux-2-pro` / `gpt-image-1.5` unless the key has the corresponding distributor channel (most pockgo keys do not).
2. OCR uses **PaddleOCR PP-OCRv5 mobile** (`PP-OCRv5_mobile_det` + `PP-OCRv5_mobile_rec`) with `enable_mkldnn=False`. Do NOT use the server variant on Windows (see Failure Modes #2). Do NOT use a vision LLM as the OCR engine (see Failure Modes #1).
3. The interactive HTML is built by `scripts/build_interactive_page.py`, which **inlines** the hotspots JSON. Do NOT `fetch()` the JSON from the page — it breaks under `file://` (see Failure Modes #3).
4. Stage dimensions in the HTML MUST come from `data.width / data.height` in the hotspots JSON, NOT from the prompt's requested size — `gpt-image-2` returns the closest supported aspect, not exact pixels (see Failure Modes #4).
5. Only big horizontal text becomes clickable (height ≥ 60px AND width/height ≥ 1.1). Small captions, vertical column labels, diagonal decorative text, and tiny chart numbers stay visual-only.

### Failure Modes Encountered (2026-05)

| # | Failure | Symptom | Root cause | Fix |
|---|---|---|---|---|
| 1 | Vision LLM as OCR | bboxes 30–50% off, half the text missed even with `max_tokens=32000` | LLMs are not localization models; coordinates are hallucinated | Switch to PaddleOCR. Pixel-accurate, deterministic, free, ~3s per page. |
| 2 | `PP-OCRv5_server_det` crash on Windows | `NotImplementedError: ConvertPirAttribute2RuntimeAttribute ... ArrayAttribute<DoubleAttribute> ... onednn_instruction.cc:118` | PaddlePaddle 3.3.1 + MKLDNN incompatibility with the server detection model on CPU/Windows | Force mobile det+rec models: `text_detection_model_name="PP-OCRv5_mobile_det"`, `text_recognition_model_name="PP-OCRv5_mobile_rec"`, `enable_mkldnn=False`. |
| 3 | HTML page goes black | Image never appears, no console error visible | `fetch("hotspots.json")` blocked by browser CORS under `file://` → main JS throws → image src never set | `build_interactive_page.py` inlines the JSON as a literal `const DATA = {...}` and sets `<img src=...>` in HTML, not via JS. |
| 4 | HTML page also goes black after fetch fix | Image present but cropped/empty | `#stage` was hardcoded to 1792×1024 but `gpt-image-2` actually returned 1659×948 for `size=1792x1024` | Read real dims from `data.width / data.height` in the JSON. Builder script does this automatically. |
| 5 | `gpt-image-2-4k` returns 524 | `HTTP 524 origin_response_timeout` after 120s | Origin generation slower than Cloudflare's 120s proxy read timeout | Drop to `gpt-image-2-2k` (native 2048x1152, ~2.7MB, ~30s). It is the practical ceiling on this relay. |
| 5b | `nano-banana-pro` / `gemini-3-pro-image-preview` / `gpt-image-1.5` return 503 `model_not_found` | `No available channel for model X under group <distributor>` | Key is bound to a distributor group that does not carry these channels | Stick to `gpt-image-2-2k`, or buy a key in the right distributor group. |
| 6 | Small Chinese typos in baked text (e.g. 现金流→现金洇, 选下一条→退下一条) | One or two characters per page off | gpt-image-2 character-level error rate ~1–2% on dense CJK | Acceptable for big-title-only interactivity (typos in small print don't affect hotspots). For correction, use `/v1/images/edits` to repaint just the bad region (not yet wrapped). |

### Image Prompt Recipe (Image-As-Page Specific)

**Use `references/image-as-page-prompt-recipe.md` as the prompt contract.** That file defines the 7 mandatory slots (palette / left typography / right mockup / corners / critical typography / negatives), 4 ready-to-fill style directions (dark-tech / warm-commercial / clean-SaaS / magazine), the density-limit table that prevents HTTP 524 timeouts at 2K, and a self-check list. The skeleton below is for orientation only; never send a prompt that hasn't been filled against the recipe.

```text
[high-craft editorial single-page composition, 16:9]

PALETTE (locked, must match other pages in this deck):
  bg, primary text, accent gradient (start, end), secondary accent, muted UI

LAYOUT:
  Left typography zone (40-50% width):
    - small label
    - massive 2-line title (line 1: gradient bold, line 2: italic serif)
    - 3-line description
    - row of KPI tiles with gradient-filled numbers
    - role chips
  Right mockup cluster (50-60% width):
    - main device (laptop / desktop browser, isometric, 12° tilt, soft shadow)
    - in-frame realistic UI (real Chinese strings, real numbers, real chart)
    - 1-2 floating secondary elements (phone, color token, comment popover)
    - small "UIUX" chip top-right of cluster

CORNERS:
  Top-right: outline-stroke "PORTFOLIO" wordmark + project subtitle
  Bottom-right: copyright line

CRITICAL TYPOGRAPHY RULES:
  - Render ALL Chinese characters crisply, no missing strokes (PingFang / Source Han Sans)
  - KPI numbers: gradient fills the GLYPH, not behind it
  - No watermark, no signature, no extra logos
  - No motion blur, no flare, no grain
  - Calm editorial composition with breathing space
```

### Page-Build Quick Recipe

```powershell
# 1. Generate the page (default model = gpt-image-2, accepts 2048x1152)
$env:POCKGO_KEY = "<key>"
python -u scripts/image_client.py generate `
  --prompt-file prompts/<page>.txt --aspect 16:9 `
  --out generated-images/demo/<page>.png

# 2. OCR -> hotspots.json
python -u scripts/ocr_hotspots_paddle.py `
  --image generated-images/demo/<page>.png `
  --out   generated-images/demo/<page>.hotspots.json --lang ch

# 3. Build self-contained interactive HTML
python scripts/build_interactive_page.py `
  --hotspots generated-images/demo/<page>.hotspots.json `
  --out      generated-images/demo/<page>.html `
  --title "<Page Title>"

# 4. (optional) Visual verify OCR
python scripts/overlay_hotspots.py `
  --image    generated-images/demo/<page>.png `
  --hotspots generated-images/demo/<page>.hotspots.json `
  --out      generated-images/demo/<page>.overlay.png
```

## Out Of Scope For This Workflow

- Editable PPTX export. If the user later asks for it, run huashu's `html2pptx.js` against the finished HTML deck — caveat: the background image becomes a single image layer in PowerPoint; only the HTML text on top remains as editable text boxes.
- Live data binding / real-time charts. Concept screens are generated images, not live components.
- Drag-to-reposition Figma-style editing. The deck is "page-turn + click links", per user decision.
