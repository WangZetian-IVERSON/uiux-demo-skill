# Quality Review

> **Pre-requisite**: every multi-zone overlay page MUST have already passed the
> per-page **closed-loop overlay QA** (`scripts/build_with_qa.py`, see
> `references/closed-loop-overlay-qa.md`) with `pass=true` and zero
> `blocker|major` issues. This document is the deck-level review that runs
> AFTER every page already passed its per-page loop. Do not use deck-level QA
> to catch image-text mismatches — that's the closed loop's job.

## Review Dimensions

Score each dimension from 1-10 before delivery.

Narrative clarity:
Can the viewer understand the project context, problem, decision path, and result without explanation?

Design reasoning:
Does each major visual decision connect to a user, business, interaction, or information problem?

Visual hierarchy:
Does every page have a clear first read, second read, and third read?

Craft:
Are alignment, spacing, screenshot framing, typography, color, and visual effects controlled?

Portfolio distinctiveness:
Does the deck feel specific to this designer/project rather than like a generic AI template?

Output reliability:
Are text, images, and charts readable in the final format? Are there no overflows, broken assets, or unreadable tiny labels?

Asset integrity:
Are real screenshots/logos used wherever the user provided them? In VISUAL_PROPOSAL mode, are concept UI screens generated by the image model (or rendered as designed dashboards via dashboard-rendering.md) AND clearly framed as concept work rather than claimed as shipped product? No HTML/CSS gray-bar fake UI saved as PNG anywhere.

Analysis depth:
Does every analysis page (problem, competitive, strategy, IA, comparison, outcome) pass the analysis-depth.md specificity bar — named segment + named scenario + counted observation? No filler phrases like "提升用户体验" / "优化操作流程".

Showcase validation:
For decks with 5+ pages, were two visually different showcase pages created and approved before batch production?

Visual richness:
Does the deck have at least three distinct page types and enough image/device/component/data proof to avoid title-and-bullets simplicity?

## Fast Fix Checklist

Use this checklist after generating a deck:

- Each project has a clear role and responsibility.
- Each project has at least one problem page and one design strategy page.
- Competitor/benchmark pages include findings, not only screenshots/logos.
- Every before/after or option A/B page explains why the chosen option wins.
- UI screenshots are framed or integrated, not pasted flat.
- Generated images are used aggressively where they belong: covers, section covers, 3D props, atmospheric scenes, posters, AND — in VISUAL_PROPOSAL mode — concept UI screens. Dashboards / data screens are HTML+SVG mockups per dashboard-rendering.md, never image-model output.
- No HTML/CSS-rendered UI presented as either a screenshot or a generated image.
- No fake metrics or user quotes appear as facts. Designed sample data inside a dashboard mockup is fine; metric claims in case-study narrative are not.
- Dense pages have enough structure.
- Hero pages have enough visual ambition.
- Decks with 5+ pages have 2 showcase pages before batch production.
- The deck includes varied page types: cover, problem/strategy, UI proof, component/data/visual system, outcome.
- Metadata is consistent across pages.
- Ending page includes contact or next action.

## Common Fixes

If the deck is pretty but shallow:
Add problem diagnosis, strategy, and tradeoff pages.

If the deck is logical but boring:
Add section covers, stronger hero mockups, spatial screenshot treatment, and more page rhythm.

If the deck feels like a simple PPT:
Stop full production, create 2 showcase pages, lock grammar, then rebuild remaining pages with varied page patterns and a richer asset pool.

If the deck feels generic:
Add real project assets, domain-specific visual cues, and specific design claims. Run analysis-depth.md self-check on every analysis page — force named segment / scenario / count.

If the deck feels empty (placeholder boxes everywhere):
This is a failure mode, not caution. Switch to VISUAL_PROPOSAL mode, generate concept UI screens via the image model and render dashboards via dashboard-rendering.md. The only acceptable placeholders are: real shipped product screens (when the project IS a real shipped product), real brand logos that don't exist yet, real user quotes, real metrics.

If the deck feels cluttered:
Reduce body copy, increase contrast, move details to callouts, and alternate dense pages with visual pages.

If the deck feels too dark-tech for the project:
Switch to clean product case study or editorial design story.
